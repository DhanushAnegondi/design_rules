# CSS gradients

> CSS gradient functions produce resolution-independent color transitions directly in the browser — no image files, no HTTP requests — and double as a pattern primitive when hard stops and `background-size` are combined.

**Bucket:** visual style
**Maturity:** evergreen
**Effort:** low
**Best for:** websites, portfolios, apps, dashboards

## What it is

A CSS gradient is an `<image>` value generated by the browser from a mathematical description of how colors transition across a space. `linear-gradient()` interpolates along a straight axis; `radial-gradient()` radiates outward from a center point; `conic-gradient()` sweeps around an angle. Each has a `repeating-` variant that tiles the defined pattern infinitely. Because the output is a vector image at render time, gradients scale perfectly at any resolution, can be layered with `background` shorthand, and respond to custom property changes without repainting the layout.

## When to use

- Large surface backgrounds (hero sections, full-page backdrops) where a flat color is too stark but a photograph is too distracting.
- Depth cues on cards, sidebars, or navigation bars — a subtle top-to-bottom linear fade reads as ambient lighting.
- Geometric patterns — checkerboards, dots, stripes, plaid — using hard stops and `background-size` instead of external images.
- Color wheels and pie-chart segments via `conic-gradient()` with hard angle stops.
- Animated brand moments: register color stops as typed `@property` custom properties and `transition` them — the only way to animate gradient colors natively without JavaScript.
- Banding-free, perceptually smooth color ramps by explicitly interpolating `in oklch` rather than accepting the default sRGB path.

## When NOT to use

- Body text backgrounds: even a subtle gradient behind multi-paragraph text creates shifting contrast at different points along the gradient, making WCAG AA compliance unpredictable and hard to test reliably.
- As the sole source of meaning (e.g., a gradient-only status indicator): gradients carry no semantic value; pair with text, icon, or ARIA label.
- Everywhere at once — this is the dominant AI-generation tell (see Anti-slop). Gradients read as designed choices only when used sparingly.
- When a photograph or illustration is available and more appropriate: a gradient faking depth is weaker than real imagery for hero moments.
- High-contrast mode: backgrounds set via `background-image` are suppressed by `forced-colors` media query — provide a `background-color` fallback.

## How it works

CSS gradient functions are declared as values for `background-image` (or the `background` shorthand). They accept an ordered list of **color stops** — each stop is a color value plus an optional position (percentage, length, or for conic gradients, an angle). The browser linearly interpolates between adjacent stops in the specified color space.

**Color stop positions** default to even spacing when omitted. Positioning two adjacent stops at identical locations creates a **hard stop** — an abrupt color boundary with zero transition. A single stop can carry two positions (e.g., `red 30% 60%`) to define a solid color band.

**Color hints** are positionless values between two stops that shift the interpolation midpoint: `linear-gradient(red, 20%, blue)` reaches the 50/50 mix at 20% rather than 50%.

**Color interpolation method** (`in <color-space>`) overrides the default color space for the interpolation path. The default for gradients is sRGB, which produces gray or muddy midtones when the two hue angles diverge significantly. Writing `in oklch` or `in oklab` routes the path through a perceptually uniform space, keeping midtones vivid.

**Layering** works by listing multiple gradient (or image) values in the `background` shorthand, separated by commas; earlier entries paint on top. Partial transparency is required for lower layers to be visible.

**Key functions and syntax:**

```
linear-gradient( [<angle> | to <side-or-corner>]?, <color-stop-list> )
linear-gradient( [in <color-space>]?, [<angle>]?, <color-stop-list> )

radial-gradient( [<shape> <size>]? [at <position>]?, <color-stop-list> )

conic-gradient( [from <angle>]? [at <position>]?, <angular-color-stop-list> )

repeating-linear-gradient( … )
repeating-radial-gradient( … )
repeating-conic-gradient( … )
```

**`@property` for animation:** CSS custom properties are treated as untyped strings by default, so the browser cannot interpolate between two gradient declarations during a `transition`. Registering each color stop as a typed `<color>` property with `@property` lets the browser animate the individual color value, making native gradient transitions possible.

### 2. Banding fix — SVG feTurbulence grain overlay

Banding and OKLCH muddy midtones are two separate problems. OKLCH fixes the muddy midtone; grain fixes 8-bit quantization banding.

```html
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>Gradient banding fix</title>
<style>
  body { margin: 0; background: #0c0c10; display: flex; gap: 2rem;
         align-items: center; justify-content: center; min-height: 100vh; flex-wrap: wrap; }

  .panel {
    width: 320px;
    height: 220px;
    border-radius: 16px;
    position: relative;
    overflow: hidden;
    display: flex;
    align-items: flex-end;
  }

  /* Gradient on a pseudo-element so the filter does NOT affect
     any child text or box-shadow on the parent */
  .panel::before {
    content: '';
    position: absolute;
    inset: 0;
    z-index: 0;
  }

  .panel-label {
    position: relative;
    z-index: 1;
    padding: 1rem 1.25rem;
    font-family: system-ui, sans-serif;
    font-size: 0.875rem;
    /* Text on darkest end: oklch(14% 0.05 264) bg → #e8e8ee text
       Contrast ratio ≈ 13.4 : 1 — WCAG AAA */
    color: #e8e8ee;
  }

  /* Without grain */
  .no-grain::before {
    background: linear-gradient(in oklch 160deg,
      oklch(55% 0.2 264),
      oklch(14% 0.05 264)
    );
  }

  /* With grain — filter applied only to the ::before pseudo-element */
  .with-grain::before {
    background: linear-gradient(in oklch 160deg,
      oklch(55% 0.2 264),
      oklch(14% 0.05 264)
    );
    filter: url(#grain);
    /* clip-path keeps the filter contained within border-radius */
    clip-path: inset(0 round 16px);
  }
</style>
</head>
<body>

  <!-- Hidden SVG filter — zero painted area, aria-hidden -->
  <svg width="0" height="0" aria-hidden="true" focusable="false"
       style="position:absolute">
    <filter id="grain" color-interpolation-filters="sRGB"
            x="0" y="0" width="1" height="1">
      <feTurbulence type="fractalNoise" baseFrequency="0.72"
                    numOctaves="4" stitchTiles="stitch"/>
      <feDisplacementMap in="SourceGraphic" scale="18"
                         xChannelSelector="R" yChannelSelector="G"/>
      <feBlend in2="SourceGraphic" mode="screen" result="blended"/>
      <feComposite in="blended" in2="SourceGraphic" operator="in"/>
    </filter>
  </svg>

  <div class="panel no-grain">
    <span class="panel-label">No grain — banding may be visible on 8-bit displays</span>
  </div>

  <div class="panel with-grain">
    <span class="panel-label">Grain overlay — banding broken up by noise displacement</span>
  </div>

</body>
</html>
```

### 3. Animated gradient via `@property` — native CSS, no JavaScript

```html
<!DOCTYPE html>
<html lang="en">
<head>
<meta charset="utf-8">
<title>Animated gradient via @property</title>
<style>
  /* Register each color stop as a typed <color> so the browser
     can interpolate them during a CSS transition */
  @property --g-start {
    syntax: "<color>";
    inherits: false;
    initial-value: oklch(55% 0.22 264);
  }

  @property --g-end {
    syntax: "<color>";
    inherits: false;
    initial-value: oklch(40% 0.18 295);
  }

  body {
    margin: 0;
    min-height: 100vh;
    background: #0f0f13;
    display: flex;
    align-items: center;
    justify-content: center;
    font-family: system-ui, sans-serif;
  }

  .btn {
    --g-start: oklch(55% 0.22 264);
    --g-end:   oklch(40% 0.18 295);
    padding: 0.875rem 2.25rem;
    border: none;
    border-radius: 8px;
    font-size: 1rem;
    font-weight: 600;
    cursor: pointer;
    /* text on gradient: lightest end is oklch(55% 0.22 264) ≈ #5b7fe0;
       white #fff on that surface: contrast ≈ 4.6 : 1 — WCAG AA */
    color: #ffffff;
    background: linear-gradient(in oklch 135deg, var(--g-start), var(--g-end));
    transition: --g-start 400ms cubic-bezier(0.16, 1, 0.3, 1),
                --g-end   400ms cubic-bezier(0.16, 1, 0.3, 1),
                transform 200ms cubic-bezier(0.16, 1, 0.3, 1);
  }

  .btn:hover,
  .btn:focus-visible {
    --g-start: oklch(65% 0.25 30);
    --g-end:   oklch(50% 0.22 55);
    transform: translateY(-2px);
    outline: 3px solid oklch(65% 0.25 30);
    outline-offset: 3px;
  }

  .btn:active {
    transform: translateY(0);
  }

  /* prefers-reduced-motion: drop the transition but keep the visual state change */
  @media (prefers-reduced-motion: reduce) {
    .btn {
      transition: none;
    }
  }
</style>
</head>
<body>
  <button class="btn" type="button">Start free trial</button>
</body>
</html>
```

**How the animation works:** Without `@property`, a CSS variable holding a color is an opaque string — the browser cannot compute an intermediate value between two strings. Registering `--g-start` with `syntax: "<color>"` gives the browser a typed value it knows how to interpolate frame-by-frame during the `transition`. The gradient re-evaluates each frame as the variable changes, producing a smooth color shift with zero JavaScript.

**Browser support for `@property`:** Chrome/Edge 85+ (2020), Safari 16.4+ (2023), Firefox 128+ (July 2024). Baseline widely available as of 2024. For older browsers the `transition` silently degrades to a snap-change; the visual state change still happens.

## Variations

| Variant | Defining knob |
|---|---|
| **Linear — angled** | `<angle>` or `to <side-or-corner>`; `45deg` diagonal is the default tell, prefer intentional angles like `160deg` |
| **Linear — in oklch** | Add `in oklch` (or `in oklab`) before the angle to route interpolation through a perceptually uniform space |
| **Radial — spotlight** | `ellipse <w> <h> at <x> <y>` positions a soft-edged light source anywhere in the element |
| **Conic — color wheel** | `in oklch longer hue` traverses all 360° evenly; `in hsl shorter hue` clips to the short arc |
| **Conic — pie chart** | Hard angle stops (`red 0turn 0.25turn, blue 0.25turn 0.6turn`) divide the circle into solid sectors |
| **Hard stop — stripe pattern** | Two stops at the same position + `repeating-linear-gradient` + `background-size` tiles stripes |
| **Hard stop — checkerboard** | `conic-gradient` with four 0.25-turn hard stops + `background-size` |
| **Dots pattern** | `radial-gradient` with a 1–3 px `circle` stop to `transparent` + `background-size` |
| **Layered depth** | Stacked `radial-gradient` layers at partial opacity composite over a solid base color |
| **Animated via `@property`** | Register stops as `<color>`, `transition` them — smooth color shift with no JS |
| **Grain-dithered** | SVG `feTurbulence`+`feDisplacementMap` filter on the gradient pseudo-element breaks up 8-bit banding |
| **Eased gradient (more stops)** | Add manually computed intermediate stops at eased positions to fake a non-linear interpolation curve |

## Accessibility

**Forced-colors / high-contrast mode:**
`background-image` gradients are suppressed in `forced-colors: active`. Always declare a `background-color` alongside so the element retains its intended surface color:

```css
.hero {
  background-color: oklch(40% 0.18 264); /* fallback for forced-colors */
  background-image: linear-gradient(in oklch 160deg, oklch(55% 0.22 264), oklch(14% 0.05 264));
}
```

**Text contrast over gradients:**
A gradient background presents different luminance at every pixel. Always measure contrast at the *lightest* point where text overlaps, not the average. Use the APCA or WCAG 2.1 formula against the specific hex at that point. In the banding-fix example above, white text (#ffffff) over the lightest gradient stop (`oklch(55% 0.2 264)` ≈ `#5e7de0`) yields approximately 4.6:1 — WCAG AA for body text. For the animated button, confirm that both color states pass before shipping.

**Semantic information must not live in the gradient alone:**
Use text, icons, or ARIA attributes to communicate status or category — gradients carry no accessible semantics.

**Pointer and touch fallback:**
The animated `@property` button uses `:hover` and `:focus-visible`. Touch devices do not fire `:hover` persistently, which is fine here — the resting state is already styled and the interaction is a nice-to-have enhancement.

```css
/* Hover-only effects explicitly scoped to fine-pointer devices
   (cursor spotlights, gradient shifts tied to mouse position via JS) */
@media (hover: hover) and (pointer: fine) {
  .cursor-reactive-gradient {
    transition: --g-start 200ms, --g-end 200ms;
  }
}
```

**Prefers-reduced-motion:**
The animated gradient button already handles this — drop `transition` entirely, keep both visual states. Nothing fades, flies, or pulses. Any JS-driven gradient animation (e.g., `requestAnimationFrame` rotating a conic gradient) must also respect this query:

```js
const reduce = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
if (!reduce) {
  // start the animation loop
}
```

**Screen readers:**
Gradient backgrounds set via CSS are invisible to assistive technology — no `aria-*` required unless the gradient encodes information visually that has no text equivalent.

## Performance

**No layout thrash, no repaints from transform/opacity:**
Gradient backgrounds paint to a GPU composited layer when the element has `will-change: transform` or is otherwise promoted. Avoid `will-change` on every element — it consumes GPU memory. Only promote elements that actually animate.

**Repaint cost:**
Changing a CSS gradient value (or the custom properties feeding one) triggers a repaint of that element's background. A repaint is cheap compared to layout, but animating a gradient at 60 fps via `requestAnimationFrame` on a large hero element is still expensive. For animated gradients, prefer the `@property` + CSS `transition` approach (browser-optimized) over JS-driven frame updates.

**SVG filter cost (`feTurbulence`):**
`feTurbulence` runs on the CPU by default and is expensive if the filtered surface is large and must re-render continuously. Apply it only to static or rarely-changing gradients. If the gradient animates, keep the filter on a separate static pseudo-element underneath and layer the animated gradient above it.

**`backdrop-filter` is unrelated but similarly costly:**
Do not confuse background gradient performance with `backdrop-filter` (blur/saturation on content behind the element) — the latter is significantly more expensive and causes a separate compositor layer.

**Gradient complexity and banding:**
More color stops generally reduce banding by giving the browser more anchor points, but they add negligible rendering cost. The grain-fix SVG filter is the larger cost — benchmark on target hardware.

**`background-size` and pattern tiling:**
Repeating gradient patterns with small `background-size` (e.g., 20px × 20px) tile hundreds of times in a large element. This is rasterized once by the GPU and cached — the cost is memory (a 20px tile replicated 500 times), not per-tile rendering.

## Anti-slop

**The cliche (see `_slop-blocklist.md` → Color):**
The most recognizable AI-generated surface: a purple-to-pink diagonal gradient (`linear-gradient(135deg, #6366f1, #ec4899)`) over a white card or hero. It signals "I grabbed the defaults." Second place: the generic SaaS blue-to-indigo gradient with no relationship to the rest of the design system.

**The tasteful alternative:**
- Commit to a single brand hue and build the gradient as a lightness/chroma ramp within that hue using OKLCH: `linear-gradient(in oklch 160deg, oklch(62% 0.2 264), oklch(32% 0.12 264))`. One hue, two luminance poles — far more restrained and more easily WCAG-compliant.
- If two hues are needed, make them tonally adjacent (no more than 60° apart on the color wheel) rather than complementary — adjacent hues create warmth, opposite hues fight each other unless handled carefully.
- Avoid gradients entirely on body copy backgrounds. Reserve a single gradient moment for the primary hero or one decorative surface.
- For patterns (stripes, dots, checkerboard), a subtle single-hue low-contrast repeat reads as deliberate texture; a multi-color rainbow stripe reads as a default placeholder.

## Pairs well with

- `noise-grain-texture` — adding a feTurbulence SVG grain overlay on top of a gradient removes banding and adds tactile depth (same technique shown in Working code §2)
- `glassmorphism` — a translucent frosted card over a blurred gradient background is the one acceptable use of layered translucency; keep to a single moment
- `cursor-spotlight-glow` — a radial gradient centered on the cursor position, driven by `mousemove` → CSS custom properties, creates a reactive ambient light effect
- `aurora-blob` — animated or static radial/conic gradients composited with `mix-blend-mode: screen` or `lighten` over a dark base simulate ambient light blobs; confine to the hero and use a single hue (see slop-blocklist → "aurora blob background")
- `@property` custom properties — enables the animated gradient variation described above; the approach extends to any typed CSS value (angle, length, color)
